# Git 多仓库管理器 row绑定与数据同步 Bug 修复需求

请先修复 Git 多仓库管理器当前存在的两个核心数据同步 Bug。

这是已有项目修复，不进行架构重构，不修改 UI 布局，只修复数据绑定逻辑。

## 一、修复 row 绑定导致的更新错乱问题

### 当前问题

仓库列表使用表格 row 作为仓库唯一标识。

删除、排序、重新扫描后，row 会发生变化，但是按钮绑定、缓存数据仍然引用旧
row。

导致：

-   点击更新错误仓库
-   点击删除错误仓库
-   右键菜单操作错误对象
-   批量操作对象错乱

## 修改要求

禁止使用 row index 作为仓库身份标识。

禁止：

``` python
lambda: update_repo(row)
```

禁止：

``` python
self.repos[row]
```

作为核心逻辑。

统一使用：

`repo_path`

作为仓库唯一标识。

例如：

``` python
update_repo(repo_path)
delete_repo(repo_path)
show_context_menu(repo_path)
```

所有仓库操作必须通过绝对路径查找仓库对象。

------------------------------------------------------------------------

## 二、修复仓库路径和表格行不同步问题

当前存在多个数据源：

-   self.repos
-   tableWidget 内容
-   repo_cache.json
-   当前选中行

删除、扫描、排序后这些数据没有同步。

导致：

-   表格显示 A 仓库
-   内部列表保存 B 仓库
-   点击按钮执行 C 仓库

------------------------------------------------------------------------

## 修改要求

建立单一数据源原则。

要求：

`self.repos`

作为唯一仓库数据源。

UI 表格只负责显示。

数据变化流程：

    self.repos
        |
        v
    refresh_repo_table()
        |
        v
    UI显示

禁止直接修改 table 后认为数据已经改变。

------------------------------------------------------------------------

## 三、增加统一刷新方法

增加：

``` python
refresh_repo_table()
```

负责：

1.  清空表格
2.  根据 self.repos 重新生成所有行
3.  重新绑定按钮
4.  更新仓库数量
5.  恢复选中状态

以下操作必须调用：

-   扫描完成
-   删除仓库
-   添加仓库
-   重命名仓库
-   排序变化

禁止局部修改某一行。

------------------------------------------------------------------------

## 四、修复按钮绑定

当前：

``` python
button.clicked.connect(
    lambda: update_repo(row)
)
```

修改：

``` python
button.clicked.connect(
    lambda checked=False, path=repo_path:
    update_repo(path)
)
```

保证删除其他仓库后按钮仍然操作正确对象。

------------------------------------------------------------------------

## 五、右键菜单修复

右键菜单不要通过 row 获取仓库。

改为使用：

``` python
Qt.UserRole
```

保存：

``` python
item.setData(
    Qt.UserRole,
    repo_path
)
```

获取：

``` python
repo_path = item.data(Qt.UserRole)
```

右键操作统一使用：

-   删除
-   更新
-   打开目录
-   忽略更新
-   重命名

------------------------------------------------------------------------

## 六、删除仓库后的同步处理

删除成功后必须：

1.  从 self.repos 删除对应 repo_path
2.  更新 repo_cache
3.  更新 ignore_update_repos
4.  调用 refresh_repo_table()
5.  重新计算：
    -   仓库数量
    -   排序编号
    -   状态统计

禁止只删除 table 某一行。

------------------------------------------------------------------------

## 七、扫描后的数据处理

扫描完成：

不要直接追加 table。

流程：

    扫描

    ↓

    生成 Repository 对象列表

    ↓

    self.repos = result

    ↓

    refresh_repo_table()

------------------------------------------------------------------------

## 八、测试要求

测试：

1.  10 个仓库，删除第 1 个，点击原第 3 个更新，必须更新正确仓库。

2.  删除中间仓库，重新排序，点击所有更新按钮，全部对应正确。

3.  扫描、删除、再次扫描，列表和数量一致。

4.  右键删除、重命名、更新，操作对象正确。
